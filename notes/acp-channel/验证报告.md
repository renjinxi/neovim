## 1. 频道完整流程：开频道、加 agent、发消息、hide/show、stop

**结论：✅ 正确**

流程追踪：

- **开频道**：`init.lua:open_bus()` → `registry:create_channel()` → `bus_mod.new()` 创建 Channel+ChannelView → `bus:open()` 初始化 session/main agent/UI。Registry 注入 `_find_main_chat_fn` 和 `_notify_main_chat_fn` 回调，消除循环依赖。
- **加 agent**：`channel.lua:add_agent()` → 创建 Chat(headless) → 设置 on_ready/on_agent_update/on_submit/on_exit_notify 回调 → Agent.new_spawned 注册到 agents 表。on_ready 回调中设 client 和 status。
- **发消息**：`channel.lua:post()` → 写消息到 messages 数组 → 写 session log → emit "message" 事件（ChannelView 渲染）→ router:route() 路由到目标 agent。
- **hide/show**：Bus facade 代理到 ChannelView，只管窗口不杀进程。show() 重建 vsplit+input split，恢复 winbar 和 autoresize。
- **stop**：`init.lua:stop_all()` → `registry:stop_all()` 遍历关闭所有 chats 和 channels。

**remove_agent + leave 流程**：`init.lua:leave_agent()` → `bus_leave()` → `channel:remove_agent()` 设 leaving=true 并从 agents 移除 → 调用 `agent.chat:stop()` 停进程。scheduler 中 `agent.leaving` guard 防止回调继续执行。链路完整。

## 2. Chat 多实例：:Acp chat claude 每次创建新实例

**结论：✅ 正确**

- `init.lua:open_chat()` → `registry:create_chat(adapter_name, opts)` 每次 new 一个 Chat 实例
- Registry 中 key 用 `adapter_name .. "_" .. os.time()` 保证唯一（同一秒内同 adapter 会冲突，但实际操作间隔远大于 1 秒）
- **不再 toggle 已有同类型 chat**——原来的 toggle 逻辑已移除，每次都是新建
- 每个 Chat 独立拥有 buf/win/input_buf/input_win/client，互不干扰

**--join 模式**：强制覆盖频道 main agent 的 client。非 join 模式只在 main 无活跃 client 时注册。`opts.join = nil` 在传给 create_chat 前删除，正确。

## 3. 持久化恢复：:Acp restore 恢复频道

**结论：✅ 正确，有一个小限制**

- `init.lua:select_bus()` → `store.list(cwd)` 列出当前 cwd 下已保存的快照
- 用户选中后 → `store.load(filepath)` 解析 JSON → `registry:close_channel()` 关闭当前频道 → `registry:create_channel()` 创建新频道 → `bus:open()` → `bus:restore_from_snapshot(snapshot)`
- `channel.lua:restore_from_snapshot()` 回填 messages（触发 ChannelView 渲染）+ 重新启动 agents

**store.save()** 在 `channel:close()` 和 VimLeavePre 时调用，保存 channel_id/agents/history 到 `stdpath("state")/acp-channels/` 目录。

**小限制**：save 时记录了 `session_id`，但 restore 时 `add_agent` 创建的是全新 session，旧 session_id 未被利用。这意味着恢复后 agent 没有之前的上下文。这是已知限制，不是 bug。

## 4. 长消息文件化：post() 超 300 字写文件

**结论：✅ 正确（已修复 UTF-8 和路径问题）**

- `channel.lua:post()` 开头检查 `#content > 300`
- 写文件到 `vim.fn.stdpath("config") .. "/notes/acp-bus/" .. self.channel_id`（已修复为绝对路径）
- 截断用 `vim.fn.strcharpart(content, 0, 50)`（已修复 UTF-8 安全截断）
- 文件名用 `HHMMSS-sanitized_from.md`，from 的非 ASCII 字符被 gsub 清除

**正常工作**：content 被替换为 "前50字… → filepath"，后续 msg 存储和 UI 渲染都用截断版本。原始完整内容保存在文件中。

**注意**：`#content` 仍然按字节计算阈值（300 字节 ≈ 100 中文字符），这是设计选择而非 bug。

## 5. idle 系统通知：scheduler 完成后 post 系统通知

**结论：✅ 正确**

- `scheduler.lua:push_to_agent()` 的 prompt 回调中，正常完成（非 cancelled/error）时：
  1. 先检查 `agent.leaving`，如果已 leaving 则 return（防止对已移除 agent 操作）
  2. 检查 agent 在 prompt 期间是否自行 post 过消息（`self_posted` 检查）
  3. 如果没自行回复：`bus:post(name, "@main 已 idle（未主动回复）")` — 走路由，main 会收到
  4. 无论是否自行回复：`bus:post("系统", name .. " 已完成", { no_route = true })` — 纯通知，不走路由

**兜底逻辑合理**：agent 用 RPC post 了消息就不会重复通知；没有 post 则发一条 "@main 已 idle" 让 main 知道 agent 做完了但没有主动汇报。系统通知 "已完成" 让频道 UI 显示状态变化。
