# 协作模式验证 2026-03-13

## 核心模式

你下任务 → main 拆解协调 → agent 执行 → agent 有疑问问 main → main 判断不了才问你

你只在关键节点拍板，不参与细节。

## 实际价值

不是替你做决策，是**降低决策的信息难度**。
脑子乱的时候，main 帮你梳理出来，你再决策。

## 今天验证结论

- 模式跑通了，方向是对的
- 工程问题（路由/并发/system prompt）基本修完
- 需要多跑几次才知道边界在哪

## 三个角度的评价

**积极**
找到了真实痛点——信息过载时决策质量下降。把"梳理+执行"外包出去，只做判断，符合人类认知边界。今天上午"确实能帮我梳理出来"是真实反馈。

**消极**
agent 执行质量是天花板。梳理的框架是它给的，你可能在它的框架里做决策而不是自己的框架。长期用可能让你对业务的直觉变弱。

**客观**
适合"任务明确但信息量大"的场景，不适合"方向本身不清晰"的场景。系统还不稳定，工程问题会干扰对模式本身的判断，需要多跑几次才能分清"模式不行"还是"是bug"。

路是对的，但别太早下结论，先跑。

## 两种协作模式的本质区别

**模式A（现在的）**
任务明确 → agent 执行 → 完成后回报。
system prompt："完成后发一次消息"
适合：清晰的执行任务

**模式B（目标模式）**
任务可以模糊 → agent 自主推进 → 遇到判断点发消息来问 → main 决策 → agent 继续。
system prompt："遇到不确定随时发消息来问，等回复再继续"
适合：需要来回确认的复杂任务，把人从细节里摘出去但 agent 不是盲目执行

**工程挑战**
模式B 需要 agent 中途发消息、等回复、接着上下文继续。现在是"一问一答"，不支持来回。

**待解决**
这是整个系统怎么用的核心问题，下次专门讨论和测试。
