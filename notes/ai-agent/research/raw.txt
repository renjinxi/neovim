  方案 B：agent_api 里加 call_llm()，nvim 侧发 HTTP

  nvim 用 vim.system 或 curl 发请求，结果返回给 agent。

  好处：API 配置完全在 nvim 侧管理，agent 不需要知道 token。
  坏处：nvim 不擅长做 HTTP client，大响应会卡 UI。

  方案 C：暴露 API 配置 + 提供一个 shell wrapper 脚本

  get_api_config() 返回配置，同时提供一个 nvim_claude_call.sh 脚本封装好 curl

  坏处：多一个脚本文件。

⏺ 总结一下现在的架构：

  agent_api.lua 现在有 10 个函数：

  基础操作：
  - get_context() / get_buffers() / get_diagnostics() — 读 nvim 状态
  - open_file() / exec_lua() / notify() — 操控 nvim

  终端管理（新增）：
  - spawn_cli({cmd, mode?, name?}) — 在 nvim 里开任意 CLI 终端
  - get_terminal_output({name?, bufnr?, tail?}) — 读终端输出
  - list_terminals() — 列出所有已开的终端
  - send_to_terminal({name?, bufnr?, text}) — 往终端发文字

  agent 的工作流就是：

  1. spawn_cli({cmd="claude", name="review"}) → 开一个 Claude 终端
  2. send_to_terminal({name="review", text="帮我 review src/main.lua\n"}) →
  给它发任务
  3. 等一会儿...
  4. get_terminal_output({name="review", tail=100}) → 读输出结果
