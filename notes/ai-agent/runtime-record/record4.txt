  两个思路：

  方案 A：改 send_to_terminal，加一个 task 模式

  调用时传 {name="test1", text="新任务描述", task=true}，send_to_terminal
  内部自动：
  1. 重置 status 文件为 pending
  2. 更新 task.md（追加新任务描述 + 回写指令）
  3. 重新注册 watcher
  4. 把带回写指令的完整文本发给终端（而不是只发裸文字）

  好处是调用方（主 AI）不用关心回写细节，跟 spawn_cli 一样简单。

  方案 B：新增 assign_task 函数

  专门用于给已有终端追加任务，和 send_to_terminal 分开。send_to_terminal
  保持纯粹的"发文字"功能，assign_task 负责完整的任务生命周期：
  1. 写新的 task.md + 回写指令
  2. 重置 status
  3. 重新注册 watcher
  4. 通过 chansend 把任务发给终端

  好处是职责分离，send_to_terminal 还能用于非任务场景（比如发个 Ctrl+C）。

⏺ 你说得对，核心问题是状态设计。现在的模型是"一次性任务"——spawn
  时写死回写指令，done 之后整个通信链就死了。但实际需求是"持续对话"，子 AI
  是活着的，随时可以接新任务。

  现在的问题：

  1. status 只有 pending → done，没有"空闲等待中"的状态。done 之后 watcher
  停了，再发任务没人监听。
  2. 回写指令是一次性的，写在 task.md 里，子 AI 执行完就忘了。后续
  send_to_terminal 发的纯文字不带回写规则。
  3. 任务和终端是 1:1 绑定的，没有"同一个终端执行多轮任务"的概念。

  更合理的状态流应该是：

  pending → running → done → idle（等待新任务）→ running → done → idle → ...

  idle 表示子 AI 还活着，随时可以接活。watcher 在 idle 状态下继续监听。

  具体来说需要改几个地方：

  - status 加 idle 状态：子 AI 完成任务后写 done，watcher 检测到 done 后通知主
  AI，然后自动把 status 改为 idle（而不是停止监听）
  - send_to_terminal 加任务模式：发新任务时自动把 status 从 idle 改回
  pending，更新 task.md 里的回写指令，watcher 继续监听
  - 回写指令做成固定规则：不是每次写在 task.md 里让子 AI 读，而是在 spawn
  时就告诉子 AI 一个永久规则——"每次完成任务都要写 result.md 和
  status"。或者更简单，把回写指令模板化，每次 assign 新任务时自动拼上

  但这里有个根本矛盾：子 AI 不是程序，它是对话式的。你没法保证它每次都记得回写规
  则，尤其是对话轮次多了之后上下文可能丢失。

  所以可能还需要一个兜底机制：主 AI
  发任务时，把回写指令也一起发过去，而不是依赖子 AI
  "记住"。这样每次都是自包含的。
